# Dify 表格问答工作流配置建议

## 目标

把当前“意图分类后直接回复意图代码”的工作流，调整为：

```text
用户输入
→ 意图识别
→ 参数抽取
→ 按意图路由到对应表检索
→ 基于检索结果生成答案
→ 直接回复
```

核心原则：查询类问题只允许基于表格检索结果回答；没有检索到明确记录时必须反问或说明无结果。

## 现有表建议分组

| 知识库/表组 | 文件 | 适用意图 |
|---|---|---|
| 需求表 | 需求表v0.2.csv | requirement_generate, project_query |
| 项目表 | 项目表v0.2.csv | project_query |
| 任务批次表 | 任务批次表v0.2.csv | task_query, project_query |
| 规则表 | 规则表v0.2.csv | rule_query, quality_query |
| 质量批注表 | 质量批注表v0.2.csv | quality_query |
| FAQ表 | FQA表v0.2.csv | faq_query, platform_query, rule_query |
| 平台操作事项表 | 平台操作事项表.csv | platform_query, faq_query |
| 供应商表 | 供应商表.csv | supplier_query |
| 工具产品表 | 工具产品表.txt | tool_query |

建议在 Dify 里不要把所有表混成一个知识库。至少拆成 6 个知识库：需求/项目/批次、质量/规则、FAQ/平台操作、供应商、工具产品。

## 意图识别节点

节点类型：LLM

输出要求：只输出一个意图代码，方便 if-else 判断。

```text
你是数据交付业务助手的意图分类器。根据用户问题，只输出一个意图代码，不要解释，不要标点，不要换行，不要加引号。

意图代码：
- project_query：项目进度、项目状态、项目风险、某需求是否已立项、项目下一步
- requirement_generate：新需求梳理、生成需求文档、需求说明书、验收标准草稿
- rule_query：标注规则、采集规范、质检规则、怎么标、规则依据
- quality_query：质量问题、缺陷、漏标、错标、返修、质量事件、典型案例
- task_query：批次、任务进度、任务效率、谁在做、是否延期、返修次数
- faq_query：试标流程、预算单、常见问题、通用流程
- supplier_query：供应商能力、产能、质量等级、推荐供应商
- platform_query：平台操作、上传数据、创建任务、申请预算、验收、结算操作
- tool_query：标注工具、工具输出格式、JSON格式、工具版本、工具使用说明
- unknown：无法判断

用户问题：{{#sys.query#}}
```

建议模型温度：0.1-0.2。

## 参数抽取节点

节点类型：LLM

```text
你负责从用户问题中抽取表查询条件。只输出 JSON，不要解释。

用户问题：
{{#sys.query#}}

意图：
{{#意图识别节点.text#}}

输出格式：
{
  "requirement_id": "",
  "project_id": "",
  "batch_id": "",
  "rule_id": "",
  "quality_event_id": "",
  "supplier_id": "",
  "tool_id": "",
  "org_id": "",
  "status": "",
  "date_range": {
    "start": "",
    "end": ""
  },
  "keywords": [],
  "missing_fields": []
}

抽取规则：
1. 保留用户原文中的 ID，例如 REQ-001、PRJ-001、BATCH-001、RULE-001、QINC-001、SUP-001、TOOL-001。
2. 如果只出现项目名、需求名、供应商名或工具名，放入 keywords。
3. 不要猜不存在的 ID。
4. 如果问题需要精确查询但没有 ID，把缺失条件写入 missing_fields。
```

## If-Else 路由

当前工作流里所有分支都连到同一个“直接回复”节点，需要改掉。

推荐分支：

| 分支条件 | 后续节点 |
|---|---|
| `project_query` | 知识检索：项目表 + 任务批次表 + 需求表 |
| `task_query` | 知识检索：任务批次表 + 项目表 |
| `quality_query` | 知识检索：质量批注表 + 规则表 + 项目表 |
| `rule_query` | 知识检索：规则表 + FAQ表 |
| `platform_query` | 知识检索：平台操作事项表 + FAQ表 |
| `faq_query` | 知识检索：FAQ表 + 平台操作事项表 |
| `supplier_query` | 知识检索：供应商表 + 项目表 |
| `tool_query` | 知识检索：工具产品表 + 规则表 |
| `requirement_generate` | LLM 需求文档生成，必要时检索需求表/FAQ表 |
| `unknown` | 反问澄清 |

## 检索设置

对 CSV 表格，建议在 Dify 知识库里开启较小切片，或者把每行转成“字段：值”的文本块后再导入。每行一个独立块最稳。

项目表行文本示例：

```text
表名：项目表
项目ID：PRJ-001
关联需求ID：REQ-001
项目名称：夜间城市道路车辆采集一期
客户组织ID：ORG-AutoDriving-01
项目类型：采集
供应商/执行方：某外部采集供应商A
计划开始：2026/7/1
计划结束：2026/7/15
当前状态：执行中
当前阶段：执行中
质量状态：正常
验收状态：未开始
结算状态：未开始
当前风险：雨天采集效率低于预估，BATCH-002实际效率90低于预估100
下一步动作：继续推进BATCH-003采集，关注雨天天气窗口
项目负责人：张运营
```

质量批注表行文本示例：

```text
表名：质量批注表
事件ID：QINC-003
关联项目ID：PRJ-002
关联批次ID：BATCH-004
关联规则ID：RULE-003
事件类型：转写错误
严重级别：P0
影响数据量：50小时
发现阶段：全检
发现日期：2026/6/25
处理动作：全量返修
当前状态：处理中
根因分析：标注员未按双声道规则区分说话人
责任方：内部标注团队
是否需要返修：是
是否典型案例：是
```

检索参数建议：
- Top K：5-8
- Score threshold：先设 0.2-0.35，观察召回后再调
- 检索模式：优先混合检索；如果 ID 查询很多，关键词检索权重需要高一些

## 最终回答节点通用 Prompt

节点类型：LLM

```text
你是数据交付业务助手。你只能根据表格检索结果回答。

用户问题：
{{#sys.query#}}

意图：
{{#意图识别节点.text#}}

抽取参数：
{{#参数抽取节点.text#}}

表格检索结果：
{{#检索节点.result#}}

回答规则：
1. 项目状态、批次状态、质量事件、日期、效率、风险、负责人、供应商、工具版本都必须来自检索结果。
2. 不要编造表中没有的项目ID、需求ID、批次ID、规则ID、质量事件ID、供应商ID、金额、日期或统计数字。
3. 如果检索结果没有明确答案，回答“当前表中没有查到明确记录”，并提示用户补充项目ID、需求ID、批次ID、规则ID、质量事件ID或时间范围。
4. 如果检索到多条记录，按项目ID/批次ID/事件ID分组回答。
5. 回答末尾列出“数据来源”，写明使用了哪些表和关键 ID。
6. 供应商、预算、结算、报价等敏感信息，只能在检索结果明确给出且用户角色允许时回答；否则提示需要权限确认。
7. 输出简洁中文，优先给结论，再给依据。
```

## 建议测试问题

1. PRJ-001 现在进展怎么样，有什么风险？
2. REQ-002 对应项目是否有质量问题？
3. BATCH-003 有没有延期风险？
4. QINC-003 为什么要返修，依据哪个规则？
5. 语音转写应该遵守什么规则？
6. 正式试标前需要准备什么？
7. 创建标注任务要上传什么附件？
8. 哪个供应商适合做图像采集？
9. BBoxTool 的输出 JSON 长什么样？
10. PRJ-003 是否可以发起结算？

## 你当前 DSL 需要删除/调整的点

1. 删除分类 Prompt 里 `tool_query：用户问工具、输出格式、JSON格式{{#context#}}` 的 `{{#context#}}`。
2. `if-else` 每个分支不要再连接同一个 `answer` 节点。
3. `answer` 节点不要再输出 `{{#分类LLM.text#}}`。
4. 每个分支先接对应知识检索节点，再接最终回答 LLM，最后再接 answer。
5. 分类模型温度降低到 0.1-0.2，回答模型可以 0.3-0.5。
