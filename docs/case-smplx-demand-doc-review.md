## 1. 文档结论

这份需求文档已经具备进入供应商执行的基础信息，但还不够完整，存在几个需要在正式放量前确认的关键点。

当前可确认：

- 标注类型：人体关键点。
- 原始数据：视频数据。
- 执行方式：视频抽帧后转为图片标注。
- 目标格式：SMPLX 格式人体关键点。
- 交付格式：JSON。
- 误差要求：关键点误差小于 3 pixels。
- 标注范围：只标注桌面前方单人的人体关键点。
- 遮挡不可见点：不标注，不需要脑补。
- 模糊、变形、损坏图片：不需要标注。
- 多人画面：只标注桌面前方操作员。

需要补充或统一：

- 示例图只有点位示意，没有完整中文点位表，建议把 0-23 点位列表补进文档正文。

## 3. 规则口径整理

### 3.0 运营判断优先级

本需求中，需求文档明显使用模板填写，部分字段不具备最终执行效力。机器人和运营在判断任务范围时应遵循以下优先级：

```text
平台实际任务数据 > 飞书对接中明确确认的信息 > 需求文档正文规则 > 模板字段/占位字段
```

因此：

- `6996 张` 是当前任务执行范围。
- `2h` 仅作为原始视频规模背景，不作为本次标注执行量。
- 不需要反复确认“是否按 6996 张执行”，只需确认“本次上传的 6996 张是否全部需要完成”。
- 如果需求方没有特别排除样本，默认平台上传的数据都需要完成；质量规则中明确无效的图片或不可见点除外。

### 3.1 有效标注范围

- 只标注桌面前方单人的人体关键点。
- 如果一张图中存在多人，只标注桌面前方操作员。
- 需要处理人体的范围为全部人体。

### 3.2 不标注情况

- 图片模糊、变形、损坏时不需要标注。
- 人体关键点显示不完全时，该不可见点不标注，不需要脑补。
- 拍摄者指甲过长、衣袖遮挡手掌、手势偏转角度过大、手指戴戒指、手腕戴手表等特殊情况，如果遮挡人体关键点，则对应关键点不标注。

### 3.3 数据质量前提

文档说明以下情况理论上不存在：

- 图片曝光严重或太黑导致看不清。
- 图片横向倒置或扭曲。

如果执行中实际遇到这些情况，建议仍按异常样本记录，交由运营确认是否跳过。

### 3.4 点位版本风险

文档版本记录写明 `V2 删除手部标注`，但后续沟通中需求方又提出：

- 增加 `22 left_hand`
- 增加 `23 right_hand`
- 两点分别位于左右掌心

因此当前存在两个可能版本：

| 版本 | 点位数量 | 描述 |
|---|---:|---|
| 文档 V2 字面版本 | 22 个或不含手部点 | 删除手部标注 |
| 沟通确认版本 | 24 个 | 保留人体 0-21 点，并增加左右掌心 22/23 |

建议正式执行前让需求方确认最终版本：

```text
请确认本次正式标注是否以 0-23 共 24 个点为准，其中 22 left_hand、23 right_hand 为左右掌心点；不再标注完整手部关键点。
```

## 5. 建议补充的输出格式说明

当前文档只写了交付格式为 JSON，不足以指导供应商和后续验收。建议补充：

```text
交付格式为平台点标注工具 JSON。

每个有效关键点需要包含：
- label：点位名称，格式为 index_joint_name，例如 0_pelvis
- x：关键点横坐标
- y：关键点纵坐标
- valid：是否有效
- order：点位顺序

不可见或被遮挡的点不输出对应点位，不需要脑补。
```

## 6. 建议补充的验收标准

建议将验收标准写得更可执行：

```text
1. 点位误差：关键点位置误差小于 3 pixels。
2. 点位完整性：可见点应标注完整，不可见点不标注。
3. 标签准确性：label 名称和 index 必须与点位表一致。
4. 左右准确性：左右以图中人物自身左右为准。
5. 单人规则：多人画面只标注桌面前方操作员。
6. 遮挡规则：遮挡或人体自身角度导致不可见的点不标注。
7. JSON 格式：输出字段满足平台解析要求。
```
